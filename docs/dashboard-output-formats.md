# Dashboard Output Formats (L1 / L2 / L3)

This document defines every output displayed on the three dashboard levels. For each dashboard item it
specifies:

- how the information is rendered;
- which component produces it;
- whether it is generated by deterministic rules, an LLM, or both;
- whether it is currently implemented or planned;
- the underlying business rules and thresholds (where applicable);
- any known differences between the implementation and the original project plan.

The three levels:

- **L1 тАФ Executive Portfolio Dashboard** тАФ the whole portfolio at a glance (health counts, who needs intervention).
- **L2 тАФ Individual Project Dashboard** тАФ one project in depth (status, deviations, risks, recommendation).
- **L3 тАФ Data Quality Dashboard** тАФ what data is missing / stale / inconsistent, and how confident the AI is.

> **Generated By** тАФ ЁЯФв formula (deterministic C#, no LLM) ┬╖ ЁЯдЦ LLM ┬╖ ЁЯФА hybrid (deterministic + LLM).
> **State** тАФ тЬЕ live ┬╖ ЁЯЯб partial (implemented, incomplete coverage) ┬╖ ЁЯРЮ known bug (implemented, wrong
> output) ┬╖ ЁЯФ╖ planned (scoped in a follow-on register).
> **Provenance** тАФ "code" = the constant running in the agent today. "plan-doc" = the value in
> `inputs/Project Plan with Activity Descriptions.md`, a **reference, not a decided spec** (the plan doc
> marks its scoring table "EXAMPLE!"). Scoring numbers ship as placeholders
> (`HealthScoringOptions.IsPlaceholder = true`) until the PMO agrees real values at kick-off.

## Formula Reference

Each formula component below uses the same headings (empty headings omitted). "Known Divergence" carries a
severity: **High** = produces a wrong/misleading output a user would act on ┬╖ **Medium** = missing capability
or structural gap (no falsehood) ┬╖ **Low** = a threshold/tuning choice differing from a plan-doc example.

### Health Scoring тАФ `HealthScoringService` + `HealthScoringOptions`
- **Purpose:** turn a project's findings into a RAG bucket + score (read-side).
- **Formula:** per area, take the worst severity тЖТ map to a number (Green 100 ┬╖ Amber 70 ┬╖ Red 30);
  weight-normalised average over the areas present тЖТ bucket; then apply override floors (worst-case).
- **Thresholds тАФ actual shipped placeholder** (`appsettings.json` тЖТ `HealthScoring`, `IsPlaceholder: true`):
  weights, severity scores, RAG bands, confidence floor, and the 4 wired override rules тАФ current numbers
  live in `poc-data-rules-v0.md` ┬з1тАУ┬з2 (kept there so client-facing numbers aren't duplicated in two docs).
- **Plan Reference:** the plan doc's "Project Health Scoring тАФ EXAMPLE!" table: weights
  20/20/15/15/15/10/5 over **7** areas; 5 named overrides incl. "key decision overdue+blocking" and
  "data confidence very low".
- **Implementation:** `HealthScoringService.cs`; numbers bound from `HealthScoringOptions`.
- **Known Divergence (Medium):** the shipped weights cover **6** of the plan-doc's 7 areas (Decision
  added, `add-decisions-agent`); only **Scope (15%)** remains unscored тАФ it renders as a display-only POC
  signal (`ScopeSkill`, see below) until the client agrees a real Scope RAG rule + weight. "Data confidence
  very low" is handled by the separate `ConfidenceFloor` check, not the override list (by design).
- **Fixed (was High):** the `critical-milestone-missed` override (Schedule тЙеRed тЖТ Amber) previously never
  fired because the Status agent rendered a missed critical milestone as a Green "due soon" finding (see
  Schedule below). The Status agent bug is fixed (L2 register #68 item 1) тАФ the override now fires
  correctly for a missed critical milestone.

### Confidence тАФ `ConfidencePolicy.cs`
- **Purpose:** how much to trust a finding, from the data-quality signal.
- **Formula:** start High; subtract for each quality problem; take the floor.
- **Thresholds:** `MissingFieldCount` >2 тЖТ тИТ2, >0 тЖТ тИТ1; age >90d тЖТ тИТ2, >30d тЖТ тИТ1; `!SourceConsistent` тЖТ тИТ1.
  Level тЙе2 High ┬╖ =1 Medium ┬╖ else Low. An LLM's self-reported confidence is capped by this.
- **Plan Reference:** "confidence level тАФ how much to trust the status" (no formula given).
- **Implementation:** `ConfidencePolicy.FromSignals`; a PRD-added POC default.

### Schedule тАФ `StatusSkill.cs` (HealthArea.Schedule)
- **Purpose:** milestone health (variance, delay, upcoming).
- **Formula:** band by days late/overdue; a recorded adverse `Status` (Missed/At Risk) floors the
  date-derived severity; a **critical** milestone (`IsCritical`) still in trouble additionally escalates
  to Red regardless of the day-band; flag milestones due within a window as informational (dated, in the
  Upcoming-milestones panel); flag dependency-on-incomplete; surface **slip** (adjusted due тИТ baseline) as
  display-only info (it does not itself raise severity).
- **Thresholds:** see `poc-data-rules-v0.md` ┬з4 (day-bands) and ┬з7 (the 28-day upcoming window).
- **Plan Reference:** milestone adherence, schedule variance, delay severity, upcoming-milestone risk
  (next 2тАУ4 weeks), "critical milestone missed тЖТ minimum Amber" override.
- **Implementation:** reads `MilestoneRecord.DueDate`, `Status`, `BaselineDate`, `IsCritical`.
- **Fixed (was High):** the agent used to ignore `Status`/`BaselineDate`/`IsCritical` (dropped by the
  parser), so a critical, missed milestone rendered as a benign green "due soon" тАФ the worst schedule
  signal on the worst project was inverted. Fixed in L2 register #68 items 1 + 2. **Open (kickoff):**
  should slip *magnitude* raise severity on its own, not just display as info? See `poc-data-rules-v0.md`
  ┬з4's open note.

### Budget тАФ `FinancialSkill.cs` (HealthArea.Budget)
- **Purpose:** forecast overrun, spend-vs-progress, total exposure.
- **Formula:** band the forecast overrun %; flag spend running ahead of progress (only when `Actual` is
  present тАФ a missing actual is a Data Quality gap, not a Financial signal); sum the exposure.
- **Thresholds:** see `poc-data-rules-v0.md` ┬з4.
- **Plan Reference:** Green тЙд+5% ┬╖ Amber >5тАУ15% ┬╖ Red >15%; "40% done / 70% burned = high risk";
  financial exposure = overrun amount.
- **Implementation:** `FinancialSkill.ExecuteAsync`; a finding fires whenever forecast > budget.
- **Known Divergence (Low):** no 5% Green tolerance тАФ any overrun >0 emits at least an Amber finding,
  where the plan-doc example allows тЙд5% as Green. (L1 register ┬з#3.) The exposure amount + currency are
  now on `MetricValue`/`MetricUnit` (#46) тАФ no longer summary-only.

### Resource тАФ `ResourceSkill.cs` (HealthArea.Resource)
- **Purpose:** over-allocation, capacity pressure, key-role gaps, concentration.
- **Formula:** band over-allocation; flag heavy-allocation-while-absent; flag capacity pressure; flag a
  missing project-manager role; band cross-project concentration (a person spread thin).
- **Thresholds:** see `poc-data-rules-v0.md` ┬з4 (over-allocation bands, concentration 5+/3тАУ4/<3).
- **Plan Reference:** concentration 5+ projects тЖТ Red, 3тАУ4 тЖТ Amber, <3 тЖТ Green; no allocation тЖТ Red;
  absence "clarify if possible".
- **Implementation:** most checks filter assignments to the current project; concentration reads
  `slice.Data.Assignments` **unfiltered** (the full portfolio) since every slice already carries it
  (`ProjectSlice.Data` is the whole `CollectedData`, filtered inline per check тАФ no new architecture was
  needed to add this). The PM-role match (`IsProjectManagerRole`) checks "Manager", "Project Management",
  a bare "Manager", or "PM".
- **Fixed (both were prior divergences):** the PM-role check used to look only for a role name containing
  "Manager", which didn't match the data value "Project Management" тАФ producing a **false** "No Project
  Manager is assigned" on projects that clearly had one. And the plan-doc's 5+/3тАУ4/<3 concentration rule
  wasn't implemented at all. Both landed (L1 register ┬з#4) тАФ concentration is rolled up portfolio-wide by
  `ScorePortfolio.KeyPersons` and shown per-project in L2's key-deviations.

### Data Quality тАФ `DataQualitySkill.cs` (HealthArea.DataQuality)
- **Purpose:** missing fields, staleness (project- and item-level), inconsistent references, cross-source
  consistency, duplicate-identity candidates, and an informational completeness grid.
- **Formula:** flag each missing mandatory field (incl. missing budget actuals); flag stale project data;
  flag a stale RAID item (per-risk/issue staleness); flag orphan references; flag a resource allocated
  with no time logged (or vice-versa, only when time data exists); score duplicate-identity candidate
  pairs; emit one informational areas-completeness grid finding per project.
- **Thresholds:** project staleness >30 days тЖТ Amber; per-risk staleness, duplicate-candidate score, and
  duplicate-signal weights are all `DataQualityOptions` (config-bound, `Validate()`d at startup, same
  pattern as `HealthScoringOptions`) тАФ current numbers in `poc-data-rules-v0.md` ┬з8. All checks emit at
  High confidence (directly observed) except duplicate candidates (Medium тАФ a heuristic, not an
  observation).
- **Plan Reference:** "no risk update in **21 days**", budget actuals missing, milestone dates not updated,
  resource plan тЙа time entries.
- **Implementation:** `DataQualitySkill.ExecuteAsync`; project staleness measured on `LastUpdated`, per-risk
  on `RaidItemRecord.LastUpdated`; consistency now covers orphan references **and**
  resource-vs-time-entries (`TimeEntryRecord`, a POC data source тАФ no real Orbit time-entries convention
  agreed yet); duplicate candidates are emitted once per pair, never auto-merge (US-2).
- **Known Divergence (Low):** the mandatory-field set for the completeness grid and the duplicate-score
  weights are our own POC defaults (`poc-data-rules-v0.md` ┬з8.2/┬з8.4) тАФ not client-agreed. Whether
  duplication is a real problem in Orbit data at all is a kickoff question (`kickoff-questions.md` ┬зI),
  not a build blocker.

### Decision тАФ `DecisionSkill` (HealthArea.Decision) тАФ тЬЕ built (`add-decisions-agent`, #45+#47)
- **Purpose:** overdue / due-soon decisions.
- **Thresholds:** overdue = `NeededBy` past the run's as-of date and `Status` not "Approved" тЖТ **Red**;
  due-soon = `NeededBy` within 14 days and not approved тЖТ **Amber**; else no finding. Each cites its
  `Decisions!row`.
- **Override:** `key-decision-overdue` (Decision тЙеRed тЖТ floor Amber) тАФ wired.
- **Plan Reference:** the Decision KPI table + the "key decision overdue and blocking" override.
- **Implementation:** `DecisionSkill` (parallel stage); `DecisionRecord` parsed from a `Decisions` sheet by
  `ExcelProjectParser`; `HealthArea.Decision` scored at weight 10 (EXAMPLE). Recovers the missed D-1002-1
  OVERDUE signal. The L1 decision-backlog **count** roll-up (`ScorePortfolio.DecisionBacklog`) and the L2
  dedicated Decisions-needed panel (owner/deadline/consequence on `MetricDetail`) are both built.

### Scope тАФ `ScopeSkill.cs` (HealthArea.Scope) тАФ display-only POC
- **Purpose:** scope-change control / creep, as a key deviation on L2.
- **Formula:** unapproved scope increase тЖТ Red creep; approved change тЖТ Amber (moved, controlled); an
  open non-increase (e.g. a removal) awaiting decision тЖТ Amber; rejected / none тЖТ nothing.
- **Thresholds:** see `poc-data-rules-v0.md` ┬з5.
- **Plan Reference:** the plan doc weights Scope at 15% but gives no RAG rule and no scope-*change* data
  (`scope-wbs.csv` is a WBS/task tree, not a change log) тАФ the single most underspecified L1/L2 area.
- **Implementation:** `ScopeSkill` (parallel stage); `ScopeChangeRecord` parsed from a POC "Scope" sheet
  (placeholder shape тАФ the real Orbit scope-change export convention isn't agreed). Rendered in L2's key
  deviations under its own heading.
- **Known Divergence (by design, not a bug):** Scope is **excluded from the health score and the
  confidence average** (`HealthScoringService` filters it out explicitly) until the client agrees a real
  rule and weight тАФ see `kickoff-questions.md` ┬зD.

### This-period progress тАФ `SummarizeProgress.cs`
- **Purpose:** "what moved forward" тАФ a run-over-run comparison for L2 panel #2.
- **Formula:** resolve a project's two most recent runs; score each via the same pure
  `HealthScoringService` (no re-analysis, no LLM); the health-score delta drives a qualitative pace label;
  diff each run's analytic findings (matched by producing agent + citation locator, reduced to worst
  severity per key) into **moved forward** (cleared/improved) and **moved backward** (new/worsened) lists.
- **Thresholds:** pace-label bands on the score delta тАФ see `poc-data-rules-v0.md` ┬з6.
- **Plan Reference:** "what moved forward this periodтАж no moving forward / very slow / medium / okay?" тАФ
  the doc gives no thresholds (a kickoff question, `kickoff-questions.md` ┬зE).
- **Implementation:** `Application/Features/Progress/SummarizeProgress`; `GET /api/projects/{key}/progress`.
  A project with only one run reports `HasPrevious: false` ("no prior run yet"), never a fabricated delta.
